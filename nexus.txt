Introdução e contexto
O que é este material, para quem foi criado e como usá-lo.
O que é o Blueprint Founder
O Blueprint Founder é um método de implementação do ecossistema Evolution. Ele foi
criado para guiar você do zero até uma operação funcional em produção — com
arquitetura clara, fluxos validados e decisões fundamentadas.
Ele não é documentação técnica crua, não é apostila, não é curso em texto. É um
sistema guiado de execução com lógica progressiva.
Este material mostra o caminho completo. Na prática, cada projeto tem variações, edge cases e
decisões que dependem do contexto. O Pro Support existe para isso — acompanhamento
contínuo com quem conhece o ecossistema por dentro.

Visão do ecossistema Evolution
Como as peças se conectam e qual o papel de cada componente.
A Evolution não é uma ferramenta isolada. É um ecossistema completo de atendimento
digital com IA, composto por serviços especializados que trabalham juntos. Pensar nela
como "uma API de WhatsApp" é subutilizar radicalmente o que ela oferece.

A comunicação entre serviços segue um padrão claro:
Frontend → Auth: login, registro, validação de token JWT
Frontend → CRM: todas as operações de negócio (conversas, contatos, inboxes)
Frontend → Core: gestão de agentes IA, API keys, ferramentas
Frontend → Processor: chat com agentes IA em tempo real
CRM → Auth: validação de tokens em cada request (cache de 20s)
CRM → Core: proxy de operações de agentes IA
CRM → Processor: envia mensagens para bots processarem
Processor → CRM: ações de agentes (transferência, mensagens internas)

Parte 1 — Meta Business Manager (Business Portfolio)
O Business Manager (agora chamado Meta Business Portfolio) é o centro de controle
de todos os ativos da sua empresa na Meta: páginas, contas de anúncio, WhatsApp
Business Accounts, contas do Instagram Business, apps e usuários.
Criar o Business Portfolio
Erro mais comum: o nome da empresa no Business Manager não bate com o nome nos
documentos oficiais. Isso é a causa nº 1 de rejeição na verificação. Use o nome legal exato,
sem abreviações.
Verificação da empresa
A verificação é obrigatória para:
Enviar mensagens de WhatsApp além do sandbox de teste
Solicitar Advanced Access para qualquer permissão de API
Usar número de telefone de produção
Onde iniciar: Business Manager → Configurações → Central de Segurança → Iniciar
Verificação.

Parte 3 — WhatsApp Cloud API
Adicionar produto WhatsApp ao app
Sandbox de teste (antes de produção)
Após adicionar o WhatsApp, você recebe:
Um número de teste (atribuído automaticamente pela Meta)
Um token temporário (válido por 24 horas)
Sandbox que pode enviar para até 5 números verificados
O token temporário expira a cada 24h e serve apenas para testes. Para produção, você
precisa de um token permanente via System User (veja abaixo).
Registrar número de produção
Vá em WhatsApp → Phone Numbers e clique em Add phone number:
Display Name: nome visível para contatos — deve refletir a marca real da empresa
Categoria: selecione a categoria do negócio e adicione descrição
Número: insira o telefone (não pode estar ativo em nenhuma conta WhatsApp)
Verificação: escolha SMS ou ligação, insira o código OTP de 6 dígitos
Após verificado, anote o Phone Number ID — use esse nas chamadas de API, não o
número real.
Importante: o número precisa estar completamente limpo — não pode estar registrado em
nenhuma conta WhatsApp (pessoal ou Business App). Se estava, delete a conta e aguarde
até 72 horas.
Gerar token permanente via System User
O token permanente é essencial para produção. Sem ele, sua integração para de
funcionar a cada 24h.
ETAPA 1 — CRIAR SYSTEM USER
Acesse Business Manager → Configurações → Usuários → System Users.
Clique em Adicionar

Mapa de decisão
Quando usar arquitetura simples (tudo em 1 servidor)?
Se você está validando o modelo, tem menos de 50 clientes ativos simultâneos, e pode tolerar
até 30 minutos de downtime em caso de problema.
Quando separar o banco de dados?
Quando o volume de conversas cresce além de 100 simultâneas, ou quando a perda de dados
não é tolerável. Banco managed (RDS, Cloud SQL, Supabase) elimina manutenção de backup e
replicação.
Quando adicionar múltiplas instâncias?
Quando o processor (IA) se torna gargalo. O evo-processor é o serviço que mais consome
recursos por executar LLMs. Escale horizontalmente o processor primeiro, antes de escalar
outros serviços.

Boas práticas de integração
Erro caro: construir integrações sem tratamento de falha. Webhooks falham, APIs ficam
fora, timeouts acontecem. Sempre tenha um plano para "e se essa chamada não
funcionar?".

EXEMPLO 2 — PULSE: RELATÓRIO AUTOMÁTICO DO DISCORD
Todo dia às 20h o agente @pulse roda a rotina pulse-daily : varre os canais do
Discord, identifica perguntas recorrentes, detecta mudanças de sentimento, extrai
highlights e gera um relatório HTML estruturado.
Skills envolvidas:

discord-get-messages — lê mensagens dos canais configurados via API
pulse-faq-sync — atualiza o FAQ público com base nas dúvidas mais
frequentes
discord-send-message — pode postar o resumo diretamente em um canal
interno
O que o relatório entrega:
Volume de mensagens por canal e comparativo com o dia anterior
Top perguntas do dia com sugestão de resposta padrão
Score de sentimento (positivo / neutro / negativo) e tendência
Membros ativos em destaque e issues que precisam de atenção
Alertas automáticos quando sentimento cai ou volume spike ocorre
Na segunda-feira, o pulse-weekly consolida os 7 dias: MAM, crescimento,
tópicos quentes da semana e ações recomendadas.
Exemplo de uso PROMPT
@pulse gera o relatório de hoje do Discord:
volume por canal, top 5 perguntas recorrentes com sugestão de resposta,
score de sentimento e qualquer issue que precise de atenção.
Depois roda o faq-sync e posta o resumo no canal #staff-updates.

Setup
Pré-requisitos: Claude Code CLI, Python 3.11+ com uv , Node.js 18+.
Instalação rápida BASH
# Opção 1 — instalador interativo (recomendado)
npx @evoapi/evo-nexus
# Opção 2 — clone manual
git clone --depth 1 https://github.com/EvolutionAPI/evo-nexus.git
cd evo-nexus
make setup
O wizard de setup pergunta: nome, empresa, timezone, idioma, qual provider de IA
usar e quais integrações ativar. No final gera automaticamente:
config/workspace.yaml — configuração central do workspace
config/routines.yaml — horários das rotinas automáticas
config/providers.json — provider de IA ativo (Anthropic por padrão)
.env — chaves de API (você preenche após o setup)
CLAUDE.md — arquivo de contexto lido automaticamente pelo Claude Code
Subir o dashboard e o scheduler BASH
# Local (macOS / Linux)
make dashboard-app # abre http://localhost:8080
make scheduler # inicia as rotinas automáticas
# VPS — o setup instala um serviço systemd automaticamente
systemctl status evo-nexus
systemctl restart evo-nexus
journalctl -u evo-nexus -f
Primeiro uso — ponto de entrada obrigatório BASH
# Abra o Claude Code na pasta do projeto e rode:
/oracle
# O Oracle entrevista você sobre o negócio, mapeia as capabilities
# do workspace para seus problemas e entrega um plano de ativação
# em fases — você nunca precisa adivinhar o próximo passo.

CAMINHO RECOMENDADO
Roadmap: do zero à produção em 7 dias
Mini roadmap para quem quer ir direto ao ponto.
DIA 1 Preparação — Ler módulos 0 e 1. Instalar Docker. Provisionar servidor de produção.
Gerar todas as chaves secretas. Configurar domínio e DNS.
DIA 2 Setup — Clonar repositório. Configurar .env . Executar setup completo. Validar todos
os health checks. Trocar senha padrão.
DIA 3 Primeiro fluxo — Criar inbox web widget. Configurar API key de LLM. Criar agente IA.
Conectar ao inbox. Testar conversa ponta a ponta.
DIA 4 Integração — Conectar canal real (Telegram ou web widget). Configurar webhooks de
saída. Testar handoff IA/humano. Ajustar instruções do agente. Se for usar WhatsApp
Cloud API oficial, inicie o processo de verificação da empresa na Meta em paralelo
desde o Dia 1 — pode levar de 5 a 21 dias úteis dependendo da análise da Meta.
DIA 5 Produção — Configurar reverse proxy + SSL. Subir em produção. Configurar backup
automático. Executar checklist de segurança.
DIA 6 Ajustes — Testar com usuários reais. Ajustar comportamento do agente IA. Refinar
regras de automação. Validar integrações externas.
DIA 7 Rollout — Executar checklist de validação final. Documentar acessos e configurações.
Treinar equipe. Abrir para operação real.
Este roadmap assume dedicação focada. Para operações mais complexas (múltiplos canais,
integrações customizadas), considere 2-3 semanas de implementação.
